# Test Infra Follow-Up (2026-03-12)

## 背景

在为 Try-On 生产链路补稳定性修复和图片格式规范化时，顺手推进了更完整的仓库验证，结果暴露出一批历史测试债。

这批问题不属于本次生产改动本身，但已经影响到：

- 组件单测可运行性
- 全量 TypeScript 检查可用性
- 测试基线对生产改动的保护能力

因此需要单开工作处理，不与当前生产修复混在一个提交里继续扩散。

## 本次已补齐的基础设施

- 安装缺失依赖：`@testing-library/dom`
- 安装缺失类型：`@types/jest`
- 在 `tsconfig.json` 中补充 `jest` / `node` / `@testing-library/jest-dom` 类型

这些改动的作用是把“环境挡板”先拆掉一部分，让真实问题暴露出来。

## 仍待处理的问题

### 1. Jest 对 `lucide-react` 的 ESM 兼容问题

现象：

- 部分组件测试在导入 `lucide-react` 时失败
- 当前 Jest 配置仍未完全覆盖该依赖的 ESM 解析问题

建议：

- 统一采用可复用的 `lucide-react` mock 方案
- 或在 Jest 配置中明确处理该依赖的 transform / mapper 规则

### 2. 测试 mock 已落后于真实业务类型

现象：

- 多个 `next-auth` 相关测试的 session mock 不再满足当前 `Session` 扩展类型
- 组件 props mock 已落后于真实组件接口

建议：

- 建立统一的测试数据工厂
- 避免在各测试文件里散落过时的手写 mock

### 3. 组件测试存在历史接口漂移

现象：

- 例如 `DashboardStats` 的测试 props 仍使用旧字段
- 登录组件、定价组件等测试与当前实现不完全匹配

建议：

- 按组件模块分批修
- 优先修和登录、试戴、计费直接相关的测试

## 建议拆分的后续工作

1. 修 Jest 组件测试运行基线
2. 修 `next-auth` / session 相关测试工厂
3. 修 Dashboard / Pricing / Login 这几组最常跑的单测
4. 最后再恢复一次可用的 `tsc --noEmit` 测试基线

## 当前结论

本次生产改动已经具备独立验证，不需要等待历史测试债全部清完才上线。

但测试基建需要单独排期，否则后续每次生产修复都会继续被历史问题干扰。
